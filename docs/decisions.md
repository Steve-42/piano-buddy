# 决策日志

## [2026-03-10] 产品定位：习惯陪伴而非教学工具
**背景：** 市场上已有大量钢琴教学产品（Yousician、Flowkey、小叶子等），但没有专注于帮助用户坚持练琴的产品
**选项：** A) 做教学工具 B) 做习惯陪伴工具
**决定：** 做习惯陪伴工具——"帮你坚持弹琴的 AI 伙伴"
**理由：** 市场空白明确；竞品差评集中在"冷冰冰"的体验；用户（包括我自己）的核心问题是"坚持不下来"而非"不会弹"
**风险：** 功能边界模糊，容易不自觉滑向教学方向

## [2026-03-10] 技术栈选择：PWA + React + IndexedDB
**背景：** 需要选择应用形态和技术栈
**选项：** A) 原生 App B) PWA C) 微信小程序
**决定：** PWA（React + TypeScript + Vite + Dexie.js）
**理由：** 最快验证；开源友好；数据全部本地存储保护隐私；跨平台
**风险：** iOS Safari 对 PWA 支持有限（通知、后台运行等）

## [2026-03-10] 音频检测方案：Web Audio API + 频谱分析
**背景：** 需要检测用户是否在弹钢琴
**选项：** A) Web Audio API 本地分析 B) 上传音频到服务器分析
**决定：** Web Audio API 本地分析
**理由：** 隐私优先；无需服务器；钢琴声频率特征明确（27.5-4186 Hz）
**风险：** 可能有误检（其他声音被识别为钢琴声），但 MVP 阶段可接受

## [2026-03-10] LLM 集成：OpenAI 兼容格式 + 本地 fallback
**背景：** 需要 AI 生成有温度的鼓励语
**选项：** A) 固定模板 B) 调用 LLM API
**决定：** 两者结合——有 API Key 时调用 LLM，否则使用内置默认鼓励语
**理由：** 用户可选择使用自己的 API Key；无 Key 时也能正常使用；支持任意 OpenAI 兼容后端
**风险：** API 费用由用户自行承担；默认鼓励语可能不够个性化
